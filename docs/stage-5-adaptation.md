## 0. Быстрый контекст проекта

- AI-агент-«верстальщик» для арбитража: скачать лендинг → очистить → **адаптировать под оффер** → проверить → залить.
- Карта проекта и навигатор «что где» — в `ARCHITECTURE.md`. Полный воркфлоу (10 этапов) — `ARCHITECTURE.md §10`.
- Очистка живёт в `src/mastra/cleaners/`. Реестр правил/констант — `src/mastra/cleaners/registry/policy.ts`.
- Запуск (WSL+nvm): `wsl.exe -d Ubuntu-24.04 -e bash -lc 'export NVM_DIR=$HOME/.nvm; . "$NVM_DIR/nvm.sh"; cd <проект> && <cmd>'`. Тесты: `npx vitest run`. Типы: `npx tsc --noEmit`.
- **Конвенции (обязательно, из PLAN.md §4):** каждый инструмент — Zod input+output, без `any`; регистрировать в `src/mastra/index.ts`; секреты только в `.env` (+`.env.example`); после каждого шага `npx tsc --noEmit` + `npx vitest run` зелёные; правки — минимальные, апстримом.
- **Кодировка:** файлы UTF-8 править только Edit/Write, не PowerShell Get/Set-Content. PHP-вебшелл-пейлоады в тестах собирать ИЗ ФРАГМЕНТОВ (`'ev'+'al'`) — иначе Defender съест файл.

## 1. Что такое этап 5 и зачем

Двухфазная модель (policy.ts + `docs/macro-and-localization-policy.md`):
- **Очистка (готово)** делает нейтральный шаблон и *строит карту макросов*, но НИЧЕГО не подставляет — потому что на очистке ещё неизвестны вертикаль и оффер.
- **Адаптация (этот этап)** подставляет продуктовые значения под конкретный оффер: картинку и название товара, переводит тексты, вшивает наши скрипты, проверяет вёрстку, готовит к заливке.

Из `PLAN.md` это шаги 9–12, 16: инструменты `translate-text`, `replace-image`, `inject-script`, `final-visual-check`, `inject-clarity`, дирижирует ими `TaskExecutor`-агент (шаг 8). Папка `src/mastra/tools/modify/` пустая — ничего не реализовано.

## 2. Ключевой факт: как работает «замена картинки/названия» (ПОДТВЕРДИТЬ у владельца)

`{_offer_value:offerimage}` и `{_offer_value:offername}` — это **серверные макросы Keitaro**: трекер сам подставляет значения из карточки оффера в момент отдачи лендинга. Значит:

> **«Заменить картинку оффера» = вписать в `src` МАКРОС-плейсхолдер, а НЕ скачивать конкретный файл.**
> Картинка лежит в инфраструктуре владельца (CloudFront), трекер раскроет макрос в рантайме.

Для универсального лендинга подстановка тривиальна — вписать готовую константу. Сложность только в том,
**куда** вписывать (детекция продуктовой картинки), особенно когда якоря-макроса нет.

⚠️ Синтаксиса `{_offer_value:...}` НЕТ в `docs/keitaro-api-docs.json` (там только URL-макросы `{subid}` и REST CRUD). Модель известна лишь из кода/доков владельца → **подтвердить (см. Вопрос 2)**.

**Важная проблема моста между этапами:** карта макросов живёт только в памяти и рендерится в `clean-report.md` как **Markdown** (не JSON). Локатор грубый (file/tag/attr/token, без уникального селектора). → Адаптации НЕ стоит адресоваться «по карте»; надёжнее **заново пройти по DOM тем же предикатом** (cheerio), что и detect-macros. (См. решение в §5.)

## 4. Реальность лендингов (ground truth для детекции)

Изучены 7 реальных лендингов в `downloads/`. Делятся на ДВА типа — это определяет сложность:

**Тип A — статическая product-card (адверториал + карточка товара)** — `1753_clean_test`, `powergummies.shop`, `zxc.balancesour.com`:
- Продуктовый пакшот стоит вплотную к CTA, часто **обёрнут в `<a href="{offer}">`**, крупный.
- Сигналы продуктовой картинки (надёжен только в КОМБИНАЦИИ): имя файла (`prod`/`brn`/`pack`), класс (`form-product__image`), соседство с CTA/`{offer}`, обёртка в `<a href="{offer}">`, крупный размер.
- `alt` НЕнадёжен (часто пустой). Название оффера: `<title>` (powergummies), текст CTA-кнопки (1753), `<h2 class="form-product__title">` (zxc) — единого селектора нет.
- ⚠️ У адверториалов `<title>` = сенсационное обещание, НЕ название товара → имя берём из блока товара/CTA, не из title.

**Тип B — квиз-SPA (Next.js/React)** — `colonbroom.com`, `le.vshred.com`, `hims`:
- Нет статического пакшота. Продукт = логотип + шаблонные плейсхолдеры цен (`{amount}`,`{bmi}`,`{currency}`). Большинство `<img>` — трекинг-пиксели 1×1.
- Механика подстановки принципиально иная; **в v1 предлагаю НЕ поддерживать** (см. Вопрос 7).

Вывод: **универсальной «одной приметы» продуктовой картинки нет** → нужен скоринг по комбинации сигналов + шаг подтверждения (человек/агент).

## 5. Предлагаемая архитектура

### 5.1. Принципы
- **Адаптация — отдельный модуль** `src/mastra/adapt/` (сестра `cleaners/`), переиспользует `cleaners/registry/policy.ts`, `cleaners/utils/html-dom.ts` (cheerio), `verify/`. Не смешиваем с очисткой.
- **Вход — «адаптационный бриф»** (так как от YouGile пока ушли в чат): структурный объект, который владелец/агент даёт на конкретный лендинг. Это единый контракт для всех под-операций.
- **Адресация по DOM, не по markdown-карте:** каждая под-операция сама находит цели в cheerio-дереве своим предикатом (для картинки — тем же, что `detect-macros`). Карта в `clean-report.md` остаётся для человека.
- **Подстановка картинки/имени = вписать макрос** `PRODUCT_IMAGE_BASE[vertical]` / `PRODUCT_NAME_MACRO` (трекер раскроет). Режим «реальный файл» — опционально, на будущее.
- Каждая под-операция — чистая функция-проход (как в cleaners) + тонкий Mastra-tool сверху. Идемпотентность: повторный прогон не ломает уже подставленное.

### 5.2. Предлагаемый бриф (контракт входа)
```ts
interface AdaptBrief {
  siteDir: string;
  vertical: 'Adult' | 'WeightLoss';           // см. Вопрос 3
  offer: {
    imageMode: 'macro' | 'file';              // по умолчанию 'macro' (трекер подставит)
    nameMode:  'macro' | 'literal';           // по умолчанию 'macro' ({_offer_value:offername})
    nameLiteral?: string;                       // если nameMode='literal'
    imageFile?: string;                         // если imageMode='file'
  };
  translate?: {                                 // см. Вопрос 4
    targetLang: string; sourceLang?: string;
    mode: 'llm' | 'pairs';
    pairs?: Array<{ find: string; replace: string; selector?: string }>;
  };
  scripts?: Array<{ preset?: 'clarity'|'fb-pixel'|'click-tracker'; src?: string; inline?: string;
                    position: 'head-start'|'head-end'|'body-end'; async?: boolean }>; // см. Вопрос 5
}
```

### 5.3. Модули и под-операции
| Модуль | Файл (предлагаемый) | Делает |
| --- | --- | --- |
| Замена картинки | `adapt/passes/replace-product-image.ts` | находит продуктовые `<img>` (Тип A: макрос-якорь → тривиально; без якоря → скоринг §4 + порог) → вписывает `PRODUCT_IMAGE_BASE[vertical]`. Корректно собирает `srcset`. |
| Замена названия | `adapt/passes/replace-product-name.ts` | вписывает `{_offer_value:offername}` (или literal) в найденные слоты названия (приоритет: блок товара/CTA → title) |
| Перевод | `adapt/passes/translate-text.ts` + tool | режим pairs (детерминированно) и/или LLM (модель агента); не трогает атрибуты/URL/макросы |
| Инжект скриптов | `adapt/passes/inject-script.ts` + `adapt/registry/script-presets.ts` | вставляет пресеты (Clarity/FB-pixel/click) по позиции; ID из `.env` |
| Оркестратор | `adapt/adapt-site.ts` → `adaptSite(siteDir, brief)` | прогоняет под-операции по брифу, копит changelog, пишет `adapt-report.md` |
| Tools (для агента) | `tools/modify/{replace-image,translate-text,inject-script}-tool.ts` | Zod-обёртки над под-операциями |
| Финальная проверка | переиспользовать `verify/verifySite()` (уже есть visual-diff с `_backup`) | после адаптации прогнать verify; diff покажет, что подставилось |

### 5.4. Оркестрация (агент)
- v1: добавить adapt-инструменты к существующему `landingAgent` (он уже делает download→clean→verify) — один агент на весь цикл. Бриф даётся в чате.
- Отдельный `TaskExecutor`-агент (PLAN.md шаг 8) — позже, если понадобится разнести роли.

### 5.5. Порядок реализации (инкрементально, каждый шаг = код + тесты + зелёный прогон)
1. **5.1 replace-product-image (макрос, Тип A с якорем)** — самый быстрый и ценный, использует готовые константы. Цель: лендинг с продуктовым `<img>` → `src = PRODUCT_IMAGE_BASE[vertical]`.
2. **5.2 replace-product-name (макрос)** — вписать `{_offer_value:offername}`.
3. **5.3 inject-script (пресеты + .env)** — Clarity первым (ID уже в `.env`), затем FB-pixel/click (нужны ID — Вопрос 5).
4. **5.4 translate-text** — pairs-режим сначала (детерминированный, тестируемый), потом LLM.
5. **5.5 эвристическая детекция продуктовой картинки/имени** (Тип A без якоря) — скоринг §4 + порог + кандидаты на подтверждение.
6. **5.6 adaptSite() оркестратор + adapt-report.md + tools + регистрация в index.ts.**
7. **5.7 связать с verify** (финальная проверка после адаптации).
- Квиз-SPA (Тип B) — отдельной вехой после v1 (Вопрос 7).

## 6. ВОПРОСЫ К ВЛАДЕЛЬЦУ / НУЖНЫЕ ДАННЫЕ (исторические — отвечены выше)

6. **Название оффера.** Вписывать макрос `{_offer_value:offername}` или реальное название? Если реальное — откуда берётся (поле брифа)?
7. **Неуниверсальные/квиз-SPA лендинги.** В v1 поддерживаем только Тип A (статическая карточка), а квиз-SPA — позже? И как подставлять картинку, когда на лендинге НЕТ макроса-якоря — эвристикой с твоим подтверждением, или ты будешь размечать вручную?
8. **Бриф.** Ок ли формат «адаптационный бриф» (§5.2), который ты даёшь агенту в чате (вертикаль, язык, скрипты)? Что добавить/убрать?
