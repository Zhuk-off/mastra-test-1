# Red Team: `utils/normalize-landing-structure.ts`

**Роль.** Запускается **первым** в `cleanSite` (до DOM-проходов). Находит главный HTML/PHP-файл,
кладёт в корень как `index.html`, раскладывает ресурсы по `css/js/images/fonts/video/audio/assets`,
переписывает ссылки в HTML и в перенесённых CSS. Радиус поражения максимальный — кривой результат
здесь ломает или подставляет всё, что идёт дальше.

**Архитектурная рамка.** Сбор и переписывание ссылок — **регэкспы по сырому HTML/CSS** (cheerio
здесь запрещён, `docs/normalize-landing-structure-fixes.md` §16). То есть самый первый и
самый «двигающий файлы» шаг — наименее устойчивый по парсингу. Многое уже укреплено (фиксы #1–#16:
атомарный `link+unlink`, `decodePathSafe`, отсечение схем в `isRelativeUrl`, чтение 64КБ при
скоринге, суффиксы коллизий, очистка пустых папок). Находки ниже — то, что осталось мимо.

---

## Находки

### [NORM-1] 🔴 Bypass/Safety · 🟥 Critical · path traversal → перенос/удаление файлов вне siteDir

- **Сценарий:** лендинг содержит `<img src="../../../../home/asus/.aws/credentials">` (или
  `.ssh/id_rsa`, `.bashrc` — пути угадываемы).
- **Сейчас:** `isRelativeUrl('../../x')` → `true` (нет схемы, не `//`, не `/`, не `#`)
  ([:146](src/mastra/cleaners/utils/normalize-landing-structure.ts:146)). Далее
  `resolve(baseDir, decodePathSafe(fsUrl))` ([:264](src/mastra/cleaners/utils/normalize-landing-structure.ts:264))
  спокойно выходит **за пределы** siteDir; `stat` проходит (файл есть) → ресурс собран →
  `moveFileUnique` делает `link(src, dest)` + **`unlink(src)`**
  ([:192](src/mastra/cleaners/utils/normalize-landing-structure.ts:192)).
- **Последствие:** произвольный читаемый процессом файл **перемещается** (с удалением оригинала!)
  в `images/` очищенного сайта — это одновременно **уничтожение данных** и **exfil**: файл попадает
  в собранный лендинг, который пользователь затем выгружает публично. Автор лендинга полностью
  контролирует HTML. То же — в CSS-ветке (`resolve(dirname(oldPath), …)`
  [:445](src/mastra/cleaners/utils/normalize-landing-structure.ts:445)).
- **Рекомендация:** после `resolve` проверять, что путь **внутри** siteDir:
  `const rel = relative(siteDir, abs); if (rel.startsWith('..') || isAbsolute(rel)) continue;`.
  Применить в обеих ветках (HTML и CSS). Заодно решить про симлинки внутри siteDir.
- **Уверенность:** подтверждено чтением кода; гварда на выход за siteDir нет нигде.

### [NORM-2] 🔴 Bypass + 🟡 Robustness · 🟧 High · PHP-стрип бьёт только главный файл и требует `?>` (нить T-5)

- **Сейчас:** `stripPhpCode` зовётся только для **главного** файла и только если `main.isPhp`
  ([:384](src/mastra/cleaners/utils/normalize-landing-structure.ts:384)); регулярка
  `/<\?(?:php|=)?[\s\S]*?\?>/gi` ([:307](src/mastra/cleaners/utils/normalize-landing-structure.ts:307))
  **требует закрывающий `?>`**.
- **Последствие (ответ на T-5):** главный `.php` с закрытыми блоками — очищается. Но мимо проходят:
  - **незакрытый PHP** (без финального `?>` — обычная практика) → не стрипается;
  - **`.html` со вставками `<?php`** (`main.isPhp` ложен по расширению) → не стрипается;
  - **ASP/JSP `<% %>`** — `stripPhpCode` их вообще не трогает; `findMainFile` их даже не
    рассматривает как кандидатов (только `.html/.htm/.php`);
  - **все вторичные** `.php`/`.html`-файлы — normalize обрабатывает только главный.
  Во всех этих случаях серверные теги доживают до `pipeline`, где `hasServerTags` → **вся
  HTML-очистка пропускается** ([PIPE-1](pipeline.md)/[DOM-1](html-dom.md)). То есть «частичная
  защита главного файла» не закрывает скип.
- **Рекомендация:** связать с PIPE-1 (защитить-и-восстановить серверные блоки вместо удаления);
  как минимум — стрипать PHP и из `.html`-файлов с серверными вставками, и обрабатывать незакрытые
  блоки (`<?php` без `?>` → до конца файла).
- **Уверенность:** подтверждено чтением кода.

### [NORM-3] 🟡 Robustness · 🟧 High · неполный сбор ссылок + переезд главного файла = битые ссылки

- **Сценарий:** главный файл лежит в `subdir/index.html` и содержит `<img data-src="hero.jpg">`
  (ленивая загрузка) или `<video poster="thumb.jpg">`.
- **Сейчас:** паттерны `collectResources`
  ([:236](src/mastra/cleaners/utils/normalize-landing-structure.ts:236)) покрывают
  `link/script/img(src,srcset)/source/video/audio/track/iframe` и `url(...)`, но **не** покрывают:
  `data-src`/`data-srcset`/`data-bg` (lazy-load — массово в лендингах), `poster`, `<use href>`
  (SVG-спрайты), `@import "x.css"` без `url()`, и **некавыченные** атрибуты (`src=logo.png`).
- **Последствие:** такие ресурсы не собираются и не переезжают. Пока главный файл в корне — относит.
  ссылка ещё валидна. Но когда главный переезжает `subdir/ → корень`, относительные пути в
  несобранных атрибутах **ломаются** (`hero.jpg` теперь резолвится от корня, а файл остался в
  `subdir/`). Классический «пожар после переезда».
- **Рекомендация:** расширить паттерны (lazy-load атрибуты, `poster`, `@import`); либо при переезде
  главного файла переписывать **все** относительные ссылки, а не только из whitelisted-паттернов.
- **Уверенность:** подтверждено чтением паттернов.

### [NORM-4] 🟡 Robustness · 🟨 Medium · переписывание ссылок контекстно-слепо

- **Сейчас:** на каждый ресурс `html.replace(/([=\(]\s*['"])${escaped}(['"])/g, …)`
  ([:423](src/mastra/cleaners/utils/normalize-landing-structure.ts:423)) заменяет совпадение
  **где угодно** в файле, не только в исходном теге.
- **Последствие:** если `rawUrl` (напр. `logo.png`) встречается в inline-JS (`var p="logo.png"`),
  в `<meta content="logo.png">`, в data-атрибуте — он тоже перепишется на `images/logo.png`,
  ломая код/данные. Якорь по кавычке+точному совпадению ограничивает радиус, но коллизии реальны.
- **Рекомендация:** при переходе на DOM (если когда-то снимут запрет §16) — переписывать по узлам.
  Пока — хотя бы ограничить замену известными атрибутами/контекстами, а не «весь файл».
- **Уверенность:** подтверждено чтением кода.

### [NORM-5] 🟡 Robustness · 🟨 Medium · `stripPhpCode` — слепое вырезание

- **Сценарий:** `<a href="<?php echo $url; ?>">`, или `<?php echo "a?>b"; ?>`.
- **Сейчас:** регулярка вырезает `<?…?>` без понимания контекста.
- **Последствие:** `href="<?php…?>"` → `href=""` (битый пустой атрибут); строка с `?>` внутри
  кавычек обрезается по первому `?>` → хвост PHP утекает в HTML как текст. Может оставить рваную
  разметку, которую потом cheerio «починит» непредсказуемо.
- **Рекомендация:** при стрипе учитывать атрибутный контекст (убирать весь атрибут, если значение
  целиком было PHP), либо — снова — модель «плейсхолдер вместо удаления».
- **Уверенность:** подтверждено чтением кода.

### [NORM-6] 🟡 Robustness · 🟨 Medium · выбор главного файла: узкий и недетерминированный

- **Сейчас:** `findMainFile` рассматривает только `.html/.htm/.php`
  ([:96](src/mastra/cleaners/utils/normalize-landing-structure.ts:96)); при отсутствии кандидата —
  `normalizeLandingStructure` **тихо возвращается** без нормализации
  ([:341](src/mastra/cleaners/utils/normalize-landing-structure.ts:341)). Скоринг эвристический;
  тай-брейк — порядок `sort` (стабильный) → зависит от порядка обхода ФС.
- **Последствие:** вход `.xhtml`/`.asp`/`.jsp`/без расширения → нормализация молча не выполняется
  (структура не приведена, но дальше pipeline всё же чистит файлы). В многостраничном архиве
  эвристика может выбрать **не тот** главный файл → реальный контент окажется не в `index.html`.
  На равных очках — недетерминированный выбор.
- **Рекомендация:** расширить список расширений; при множестве близких кандидатов — логировать
  выбор в отчёт; стабилизировать тай-брейк (например, по relPath).
- **Уверенность:** подтверждено чтением кода.

### [NORM-7] 🟩 Low · мелочи

- В новый HTML пишется **раскодированный** путь с литеральным пробелом (`images/my logo.png`) —
  осознанное решение (§17), но некоторые серверы/CDN не отдадут URL с пробелом без кодирования.
- `readHead` декодирует первые 64КБ как UTF-8 — для UTF-16/иной кодировки скоринг-регулярки
  (`<title>`, `{offer}`) могут не сматчиться → мисскоринг главного файла.
- `@import url(...)` переписывается, а `@import "x.css"` (без `url()`) — нет; вложенные CSS-ссылки
  через `@import` не разворачиваются.

## Сильные стороны (зафиксировать)

Атомарный перенос `link+unlink` с `EXDEV`-фолбэком и суффиксами коллизий; `decodePathSafe`;
generic-отсечение URI-схем; чтение головы файла при скоринге; очистка пустых директорий. Это
закрывает целый пласт прошлых багов — находки выше про **оставшиеся** края, а не про регресс.

## Пробелы в тестах

Нет тестов на: path traversal `../` (NORM-1), незакрытый PHP / PHP в `.html` / ASP (NORM-2),
переезд главного файла с lazy-load/`poster`/`@import` (NORM-3), контекстно-слепую замену в inline-JS
(NORM-4), порчу разметки `stripPhpCode` (NORM-5), не-`.html` вход и тай-брейк (NORM-6).

## Итог

1. **NORM-1** — Critical: гвард «путь внутри siteDir» (несколько строк, закрывает exfil/удаление).
2. **NORM-2** — закрыть скип серверных тегов вместе с PIPE-1 (главный фикс этой нити).
3. **NORM-3** — расширить сбор ссылок, иначе переезд главного файла = битые ленивые картинки.
