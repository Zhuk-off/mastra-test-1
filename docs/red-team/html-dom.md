# Red Team: `utils/html-dom.ts`

**Роль.** Единственная точка входа в DOM: `parseHtml` (cheerio.load), `serializeHtml` (`$.html()`),
`hasServerTags`. В исходниках зовётся только из `pipeline.ts`. Файл крошечный (14 строк), но это
фундамент под **каждым** HTML-проходом — поведение здесь умножается на все 17 проходов.

**Эмпирика.** Гипотезы ниже проверены прогоном на cheerio из проекта (микроскрипт `rt-check.mjs`,
удалён после прогона). Вывод вставлен в находки — это факты, не предположения.

## Находки

### [DOM-1] 🔴 Bypass + 🟡 Robustness · 🟥 Critical · `hasServerTags` отключает ВСЮ очистку по ложному сигналу

- **Сейчас:** `hasServerTags` = `/<\?(php|=|\s)/i.test(html) || /<%[^>]/.test(html)`
  ([:13](src/mastra/cleaners/utils/html-dom.ts:13)). В `pipeline.ts` при `true` →
  `applyHtmlPasses` возвращает HTML **без изменений** + один лог `SKIP_DOM`
  ([pipeline.ts:79](src/mastra/cleaners/pipeline.ts:79)).
- **Замеры (true = вся HTML-очистка пропущена):**
  ```
  false  фрагмент <div>…</div>
  false  без doctype/html (<title>+<p>)
  true   <div><%= user.name %></div>        ← EJS/Underscore-вставка
  true   <script type="text/template"><% if(x){ %>…   ← шаблон в <script>
  true   текст:  cheap price <? maybe        ← обычный ТЕКСТ с "<? "
  false  <?xml version="1.0"?>
  ```
- **Последствие (двойное):**
  - 🔴 **Тривиальный обход.** Я кладу в лендинг `<!-- <% --> ` или просто текст `<? ` — и **вся**
    HTML-очистка молча выключается: трекеры, exfil, редиректы, макросы, CSP — ничего не применяется,
    файл пишется как есть. Атакующий контролирует вход. Это **самый дешёвый обход во всей системе.**
  - 🟡 **Ложные срабатывания на чистом лендинге.** Любой лендинг с шаблонами `<% %>`/`<%= %>`
    (частый билдерный синтаксис) или со строкой `<? ` где угодно — не чистится вообще.
- **Рекомендация:** перестать «пропускать всё по флагу». Варианты по возрастанию надёжности:
  (1) сузить детект до реальных серверных тегов (`<?php`, `<?=`, `<%[=@]?\s` с закрытием `?>`/`%>`),
  (2) **защитить-и-восстановить**: заменить серверные блоки на плейсхолдеры → распарсить → почистить
  → вернуть блоки, (3) для непарсимых — карантинить файл и **громко** падать в статусе, а не один
  лог в changelog. Парный фикс — [PIPE-1](pipeline.md).
- **Уверенность:** подтверждено замером.

### [DOM-2] 🟡 Robustness · 🟨 Medium · cheerio фабрикует структуру (твой кейс «нет html/doctype»)

- **Сейчас:** `cheerio.load(html)` достраивает отсутствующие `html/head/body`.
- **Замеры:**
  ```
  IN : <div class="x">hi</div>
  OUT: <html><head></head><body><div class="x">hi</div></body></html>

  IN : <title>t</title><p>x</p>
  OUT: <html><head><title>t</title></head><body><p>x</p></body></html>
  ```
- **Последствие:** чистильщик **переписывает структуру каждого HTML-файла**, даже без единой угрозы:
  фрагмент/partial-include превращается в полный документ; элементы перемещаются (title → head).
  Для цельного лендинга обычно безвредно (браузер делает то же), но: (а) include/partial ломается;
  (б) диффы «очистки» шумят структурными правками, маскируя реальные. Прямо отвечает на твоё
  опасение про отсутствующий `<html>`/doctype — cheerio их **выдумывает**.
- **Рекомендация:** определить политику осознанно. Если вход — фрагмент (нет `<html>`/doctype),
  либо сохранять как фрагмент, либо явно фиксировать в отчёте «структура нормализована». Решить
  вместе с `normalize-landing-structure`.
- **Уверенность:** подтверждено замером.

### [DOM-3] 🟢 Soundness (FN) · 🟨 Medium · содержимое `<noscript>` — это ТЕКСТ, не DOM

- **Замеры:** для `<noscript><img src="//track.er/p?fb"></noscript>`:
  ```
  noscript img count (DOM): 0
  noscript children count:  0
  noscript .text():        "<img src=\"//track.er/p?fb\">"
  ```
- **Последствие:** DOM-селекторы **не видят** трекеры/пиксели внутри `<noscript>` — они лежат
  единым текстовым узлом. Проход `removeNoscriptTrackers`, если работает через `$('noscript img')`
  и т.п., пропустит payload. Классический FN: facebook/GA-пиксель в `<noscript>` переживёт очистку.
- **Рекомендация:** в `removeNoscriptTrackers` парсить **текстовое** содержимое noscript отдельно
  (вложенный `cheerio.load`) или грузить документ так, чтобы noscript разбирался как DOM. Сверить —
  нить **T-6**.
- **Уверенность:** подтверждено замером; реальный эффект — за `removeNoscriptTrackers` (Тир 2).

### [DOM-4] 🟡 Robustness/Correctness · 🟨 Medium · `</script>` в строке ломает round-trip

- **Замеры:**
  ```
  IN : <script>var a="</script>"; doEvil();</script>
  OUT: <html><head><script>var a="</script></head><body>"; doEvil();</body></html>
  ```
- **Последствие:** парсер закрывает `<script>` на литеральном `</script>` внутри строки; хвост
  `"; doEvil();` утёк в `<body>` как текст. Это поведение HTML по спецификации, но **round-trip
  парс→сериализация делает его деструктивным**: легитимный (особенно минифицированный) скрипт со
  строкой `</script>` корёжится. Для нас скорее robustness-риск (портим валидный код), чем находка
  атаки.
- **Рекомендация:** помечать в отчёте файлы, где после сериализации изменился сырой текст `<script>`
  (сигнал порчи), либо детектить `</script>`/`</style>` в текстовых литералах до парса.
- **Уверенность:** подтверждено замером.

### [DOM-5] 🟢 Сильная сторона (зафиксировать)

cheerio/parse5 чинит незакрытые и смещённые теги — ровно то, ради чего проект ушёл от regex и
node-html-parser (память: «теги часто не закрыты → cheerio чинит структуру»). Это правильная основа;
находки выше — про **края** этого поведения, а не про замену движка.
