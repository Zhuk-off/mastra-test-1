# Red Team: Тир 5a — utils-инфраструктура

- [`walk.ts`](src/mastra/cleaners/utils/walk.ts) — рекурсивный обход файлов
- [`quarantine.ts`](src/mastra/cleaners/utils/quarantine.ts) — контейнмент (вырезать → `_quarantine/`)
- [`changelog.ts`](src/mastra/cleaners/utils/changelog.ts) — `clean-site-changes.log`
- [`report.ts`](src/mastra/cleaners/utils/report.ts) — `clean-report.md` (главный «ассистент выносит решение человеку»)

**Контекст:** вся стратегия снижения FP в системе — «не удаляем тихо, выносим человеку в отчёт +
карантин». Значит **корректность отчёта критична**: если он что-то не показывает, человек это не
проверит. Большинство destructive-действий (delete файла по эвристике из Тир 3) опираются именно на
этот ревью.

## report.ts

### [REP-1] 🟡 Soundness/Workflow · 🟨 Medium · отчёт под-показывает destructive-операции (баг типа + нет списка удалений)

- **Баг типа:** `detect-php-backdoors` пишет `type: 'PHP_BACKDOOR_WARN'`
  ([detect-php-backdoors.ts:29](src/mastra/cleaners/passes/php/detect-php-backdoors.ts:29)), а отчёт
  фильтрует warnings по `'PHP_BACKDOOR'` (без `_WARN`)
  ([report.ts:96](src/mastra/cleaners/utils/report.ts:96)) → **детали бэкдора (паттерн/файл/строка)
  в раздел warnings НЕ попадают.** Остаётся лишь общий баннер `stats.phpBackdoorWarning` без подробностей.
- **Нет списка удалений:** типы `OBFUSCATED_JS` / `METRIC_FILE` / `DEAD_JS_FILE` не входят в `warnTypes`
  и не выводятся отдельной секцией — только **счётчики** в «Итог» (`metric удалено 3`). А это самые
  FP-склонные и необратимые действия ([CJS-2](3a-js-orchestration.md)/[OBF-1](3c-detectors.md)/
  [MET-1](3c-detectors.md)/[COV-1](3d-coverage-visual.md)). Пользователь видит «удалено 3», но **не
  какие 3** → не может поймать ложное удаление. Данные есть в `changelog.log`, но не в отчёте.
- **Последствие:** главный safety-net (ревью отчёта) не показывает ни деталей бэкдора, ни *какие
  файлы удалены* — то есть именно то, что критично проверить.
- **Рекомендация:** исправить тип на `'PHP_BACKDOOR_WARN'`; добавить секцию «Удалённые файлы»
  (obfuscated/metric/dead — с путями) — раз уж удаляем по эвристике, показать что именно.
- **Уверенность:** подтверждено сверкой строк (тип vs фильтр; warnTypes vs типы удалений).

### [REP-2] 🟡 Robustness · 🟩 Low-Med · warnings обрезаны до 100

`warnings.slice(0, 100)` ([:100](src/mastra/cleaners/utils/report.ts:100)) — при >100 предупреждений
остальные не показываются. На тяжёлом лендинге реальное предупреждение может уйти за обрез.

### [REP-4] 🟢 Soundness/UX · 🟩 Low · «всё классифицировано однозначно» при пустом карантине вводит в заблуждение

Пустой карантин печатает «_Пусто — всё классифицировано однозначно._» ([:46](src/mastra/cleaners/utils/report.ts:46)).
Но пусто бывает и когда файл был `SKIP_DOM` (PIPE-1) или детекторы (слабые, DET-1/2/3) ничего не
нашли — это не «чисто». Фраза даёт ложную уверенность. Рекомендация: смягчить формулировку, учитывать
наличие SKIP_DOM/непарсимых.

## quarantine.ts

### [QUAR-1] 🟡 Robustness · 🟨 Medium · артефакты очистки лежат ВНУТРИ siteDir → уезжают в прод

- **Сейчас:** `_quarantine/` ([:47](src/mastra/cleaners/utils/quarantine.ts:47)), `clean-report.md`,
  `clean-site-changes.log` пишутся в `siteDir`.
- **Последствие:** при выгрузке очищенного лендинга эти файлы **деплоятся вместе с сайтом** и
  **доступны по URL**: `site/_quarantine/001-script.txt` (вырезанные payload'ы), `site/clean-report.md`
  (карта офферов, домены вашей инфраструктуры из CSP, что удалено), `clean-site-changes.log`. Это
  утечка: и провенанс/инфра владельца, и подсказка противнику что было удалено.
- **Рекомендация:** писать артефакты **рядом** с siteDir (как `_backup`), а не внутрь; либо явно
  исключать их при выгрузке; либо складывать в каталог вне веб-корня.
- **Уверенность:** подтверждено (пути через `join(siteDir, …)`).

### [QUAR-2] 🟢 Soundness · 🟩 Low-Med · сниппет карантина обрезан (2000 симв.) — это превью, не бэкап

`quarantineNode` сохраняет `$.html(el).slice(0, 2000)` ([:30](src/mastra/cleaners/utils/quarantine.ts:30)).
Большой вырезанный узел обрезается → «верните вручную» из `_quarantine/` невозможно для крупного
фрагмента. Полное восстановление — только из `_backup` (сайт-бэкап). Стоит явно сказать в отчёте,
что восстановление — из бэкапа, а `_quarantine` — превью для ревью.

## walk.ts

### [WALK-1] 🟡 Robustness · 🟨 Medium · нет изоляции ошибок обхода

`readdir` без try/catch ([:5](src/mastra/cleaners/utils/walk.ts:5)) — нечитаемая подпапка (права) →
исключение → генератор бросает → `cleanSite` падает целиком ([PIPE-3](pipeline.md)). Нет лимита
глубины. Рекомендация: try/catch на `readdir` (пропускать недоступное с логом).

### [WALK-2] 🟢 Сильная сторона + 🟩 Low FN · симлинки не следуются

`Dirent.isDirectory()/isFile()` для симлинка дают false → запись **пропускается** → walk не ходит по
симлинкам (нет escape-через-симлинк при обходе — хорошо). Обратная сторона: симлинкнутый ресурс не
чистится (минорный FN; для скачанных лендингов симлинки маловероятны). Не путать с
[NORM-1](normalize-landing-structure.md) — там traversal через `resolve`+`stat`, не через walk.

### [WALK-3] 🟩 Low · не исключает свои выходные каталоги

Walk не исключает `_quarantine/` явно, но его файлы — `.txt`/`.md`/`.log` → отсекаются фильтром
расширений в `pipeline` (не html/js/css/svg). На повторном прогоне безвредно. `_backup` — сосед
(вне siteDir), не задевается. Низкий приоритет.

## changelog.ts

### [CHG-1] 🟩 Low · pipe-формат ломается на `|`/переносах в сниппете

Строка `file | line | type | desc | code` ([:10](src/mastra/cleaners/utils/changelog.ts:10)) —
`codeSnippet` с символом `|` (частый в JS: `||`) или переносом строки рушит колонки/строки. Косметика
лога (+ файл уезжает в прод, см. QUAR-1).
