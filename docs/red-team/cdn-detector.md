# Red Team: `utils/cdn-detector.ts` (+ `registry/cdn-libraries.ts`)

**Роль.** Строит карту замен «исходный URL → официальный CDN + SRI». Два пути: `identifyLibrary`
(известная библиотека по имени файла, реестр `cdn-libraries.ts`) и `genericCdnRepin` (репин по
СТРУКТУРЕ пути cdnjs/jsdelivr). SRI считается стримом из официального файла (`fetchOfficial`,
sha384, лимит 5МБ, таймаут 4с, LRU-кэш 1000). Вызывается в `pipeline` на **каждый** HTML до
DOM-проходов — сюда сходятся нити **AL-3** и **PIPE-6**.

**Дизайн-рамка.** Прошлая итерация уже укреплена (комментарии в файле): bare-host `host/<pkg>@<ver>`
паттерн убран (dependency-confusion), матч только по pathname с якорем `^`, существование URL
проверяется фетчем. Находки — про оставшиеся векторы.

---

## Находки

### [CDN-1] 🔴 Bypass · 🟧 High · генерик-репин ОТМЫВАЕТ чужой хост в trusted jsdelivr

- **Сценарий:** `<script src="https://evil-host.tld/gh/attacker/payload@1.0.0/x.js">` (или
  `/npm/attacker-pkg@1.0.0/x.js`). Путь в URL уже называет репозиторий/пакет, который атакующий
  контролирует.
- **Сейчас:** `classifyResource` для `evil-host.tld` дал бы **quarantine**. Но раньше отрабатывает
  `genericCdnRepin`: pathname `/gh/attacker/payload@1.0.0/x.js` матчит структуру `jsdelivr-gh`
  ([:44](src/mastra/cleaners/utils/cdn-detector.ts:44)) → строит
  `https://cdn.jsdelivr.net/gh/attacker/payload@1.0.0/x.js` → `fetchOfficial` (атакующий опубликовал
  репо на GitHub → файл существует) → ok → репин + SRI. Затем allowlist видит `cdn.jsdelivr.net` →
  **keep**. Тест [:60](src/mastra/cleaners/utils/__tests__/cdn-detector.test.ts:60) фиксирует это как
  желаемое.
- **Последствие:** чистильщик **сам мигрирует** payload атакующего с непроверенного хоста на
  доверенный jsdelivr/gh, считает SRI и помечает доверенным — то есть **повышает** ресурс из
  quarantine в keep. SRI тут бессмыслен: он пинит контент атакующего к контенту атакующего (защищает
  лишь транзит, не происхождение).
  - Нюанс точности: **cdnjs**-структура (`/ajax/libs/...`) относительно безопасна — cdnjs курируем,
    туда не залить произвольное. Опасны именно **`/npm/<pkg>@<ver>`** (любой npm-publish) и
    **`/gh/<user>/<repo>@<ver>`** (любой github-репо).
- **Рекомендация:** генерик-репин разрешать только для **известных имён** библиотек (свести с
  `identifyLibrary`), а `/gh/` и произвольные npm-пути → quarantine. Не путать «URL существует» с
  «контент доверенный». Парно с [AL-3](allowlist.md)/[POL-2](policy.md).
- **Уверенность:** подтверждено чтением кода + тестом.

### [CDN-2] 🟢 Soundness/рамка · 🟨 Medium · «репин + SRI» создаёт ложное чувство защиты

- **Суть:** SRI гарантирует «отданный файл == тот, что мы хешировали», но **не** «контент
  безопасен». Для **hard-пути** (известная либа → канонический URL) это ок: канонический URL — это
  настоящая библиотека. Для **generic-пути** SRI пинит то, что существует (см. CDN-1).
- **Рекомендация:** задокументировать границу в `policy.ts`/отчёте: репин+SRI = «зафиксировали
  конкретную версию известной библиотеки», а не «обезопасили любой внешний скрипт».
- **Уверенность:** анализ архитектуры.

### [CDN-3] 🟢 Soundness · 🟨 Medium · версия извлекается из ПЕРВОГО токена URL

- **Сейчас:** `extractVersion` берёт первое `[@/-]<num.num[.num]>` во всём URL
  ([cdn-libraries.ts:24](src/mastra/cleaners/registry/cdn-libraries.ts:24)).
- **Последствие:** для `https://host/1.0/libs/jquery-3.6.1.min.js` первым матчится `/1.0` → версия
  `1.0`, не `3.6.1`. Дальше `getCdnUrl(1.0)` → репин на **не ту** версию (сработает только если такая
  версия есть на CDN; иначе `fetchOfficial` 404 → не репинит). Возможна замена на несовместимую
  версию → поломка API.
- **Рекомендация:** извлекать версию из сегмента **имени файла**/матча библиотеки, а не из всего URL.
- **Уверенность:** подтверждено чтением кода.

### [CDN-4] 🟡 Robustness · 🟨 Medium · кэш фиксирует негативный результат на весь прогон

- **Сейчас:** `sriCache` кеширует и `ok:false` (офлайн/404/таймаут/превышение лимита)
  ([:108](src/mastra/cleaners/utils/cdn-detector.ts:108), [:126](src/mastra/cleaners/utils/cdn-detector.ts:126)).
- **Последствие:** разовый сетевой сбой по URL кешируется как «недоступно» до конца прогона → эта
  библиотека не репинится даже после восстановления сети → остаётся локальной/в карантине. Плюс
  поведение очистки зависит от сети (PIPE-6): офлайн-прогон и онлайн-прогон дают **разный** результат.
- **Рекомендация:** не кешировать транзиентные ошибки (или короткий TTL); различать «точно нет
  файла» (404) и «ошибка сети».
- **Уверенность:** подтверждено чтением кода.

### [CDN-5] 🟩 Low · осиротевший локальный файл после hard-репина

- **Сейчас:** hard-путь (`identifyLibrary`) репинит ссылку в HTML на CDN, но локальный файл **не
  удаляет** (в отличие от unversioned-пути, где `pipeline` чистит `unversionedLibFilesToDelete`).
- **Последствие:** неиспользуемый локальный файл едет в собранный сайт. Если это был зловред,
  названный `jquery-3.6.1.min.js` (репин по имени, без проверки содержимого) — копия зловреда
  остаётся на диске (без ссылки, не исполняется, но лежит).
- **Рекомендация:** единообразно удалять осиротевшие репинутые локальные файлы.
- **Уверенность:** подтверждено чтением кода.

### [CDN-6] 🟩 Low · regex-сбор ссылок

`collectResourceUrls` ловит только **кавыченные** `script[src]`/`link[href]`; некавыченные/иные
теги пропускаются → просто не репинятся (уходят в локаль/карантин — fail-safe). `redirect:'follow'`
в `fetchOfficial` — SRI считается с конца цепочки редиректов (для версионных URL контент неизменен →
ок).

## Пробелы в тестах

Тесты есть на распознавание и на отсечение dependency-confusion/query/non-anchored — хорошо. Нет:
CDN-1 в полном потоке `buildCdnReplacements` (untrusted → keep), CDN-3 (версия из path-сегмента),
CDN-4 (кэш негатива), оффлайн-ветка `fetchOfficial`.

## Итог

1. **CDN-1** — High: генерик-репин не должен отмывать произвольные npm/gh; свести к известным именам.
2. **CDN-2** — переосмыслить «SRI = безопасность» в документации/отчёте.
3. **CDN-3/CDN-4** — корректная версия и не-кеширование сетевых сбоев.
