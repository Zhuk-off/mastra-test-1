# Red Team: `pipeline.ts`

**Роль.** Оркестратор очистки: `cleanSite()` + `applyHtmlPasses()` + `createBackup()`. Запускает
`normalizeLandingStructure` → пре-скан unversioned → обход всех файлов (HTML/PHP через DOM-проходы,
JS/CSS/SVG — свои чистильщики) → пост-обработка (удаление metric/обфусцированных/unversioned файлов
и dead-code) → карантин/changelog/отчёт.

**Сильные стороны (зафиксировать):** бэкап до очистки (в туле); `normalize` первым; `onUncertain =
quarantine`; advanced включён по умолчанию; PHP-скан на оригинальном `before` до мутаций. Находки
ниже — про края и про два архитектурных шва.

---

## Находки

### [PIPE-1] 🔴 Bypass · 🟥 Critical · серверный тег → вся HTML-очистка пропущена

- **Сейчас:** `if (hasServerTags(html))` → `applyHtmlPasses` логирует `SKIP_DOM` и возвращает HTML
  **без изменений** ([:79–86](src/mastra/cleaners/pipeline.ts:79)). Все 17 DOM-проходов не
  выполняются.
- **Последствие:** см. [DOM-1](html-dom.md) — триггерится тривиально (`<% %>`, текст `<? `, любой
  `<?php`-комментарий), и тогда трекеры/exfil/редиректы/макросы/CSP **не применяются вовсе**. Плюс
  любой легитимный `.php`-лендинг получает по сути нулевую HTML-очистку (работают лишь пост-обработка
  и отдельные JS/CSS/SVG-проходы). Предупреждение — лишь строка в changelog, статус «очищено» не
  меняется → можно молча выгрузить грязный лендинг.
- **Рекомендация:** защитить-и-восстановить серверные блоки (плейсхолдеры → парс → чистка → возврат);
  либо, если файл непарсим, **карантинить весь файл и падать громко** (выставлять флаг в `CleanStats`,
  как `phpBackdoorWarning`). Не оставлять «тихий SKIP».
- **Уверенность:** подтверждено чтением кода + замером (DOM-1). Открытая нить **T-5**: снимает ли
  `normalizeLandingStructure` PHP с **главного** файла до обхода (`NormalizeStats.phpStripped`)? Если
  да — главный index спасён, но вторичные `.php` и остаточные вставки всё равно отключают очистку.

### [PIPE-2] 🟡 Robustness + 🔴 Bypass · 🟧 High · удаление `<script src>` регэкспом мимо DOM

- **Сейчас:** после удаления metric/обфусцированных/unversioned/dead-файлов ссылки на них
  вырезаются из HTML регуляркой, **4 почти одинаковых блока**
  ([:301](src/mastra/cleaners/pipeline.ts:301), [:325](src/mastra/cleaners/pipeline.ts:325),
  [:353](src/mastra/cleaners/pipeline.ts:353), [:406](src/mastra/cleaners/pipeline.ts:406)):
  ```
  <script[^>]*\bsrc\s*=\s*["'](?:\./)?/?${escaped}["'][^>]*>\s*</script>
  ```
- **Последствие:** регулярка матчит только точный путь в кавычках. Промахи →
  **висячая ссылка на удалённый файл** (404) или несогласованная очистка:
  - query-string: `src="metric.js?v=3"` — после `${escaped}` стоит `["']`, `?v=3` не матчится → промах;
  - self-closing `<script .../>` — нет `</script>` → промах;
  - путь иной формы: `../js/metric.js` из подпапки (есть только `(?:\./)?/?`) → промах;
  - нестандартные кавычки/их отсутствие, атрибуты с `>` → промах.
  Плюс это **возврат к regex-хирургии по HTML**, от которой проект осознанно ушёл в пользу cheerio
  (память/комментарий в `html-dom.ts`). Тот же файл, что аккуратно прошёл DOM, в конце правится сырой
  регуляркой.
- **Рекомендация:** выносить удаление ссылок в **DOM-проход** по резолвнутому пути (cheerio находит
  `<script>` по нормализованному `src`, удаляет узел). Один проход вместо 4 копий регекспа. Также —
  чистить `<link>`/`<img>` на удалённые файлы (сейчас только `<script>`).
- **Уверенность:** подтверждено чтением кода; конкретные промахи — прямое следствие формы регулярки.

### [PIPE-3] 🟡 Robustness · 🟨 Medium · нет изоляции ошибок по файлу/проходу

- **Сейчас:** цикл проходов в `applyHtmlPasses` и обход файлов в `cleanSite` — без `try/catch`. Любой
  проход, бросивший исключение на одном кривом файле, роняет **весь** `cleanSite`.
- **Последствие:** один патологический вход прерывает очистку всего сайта на полпути; внутреннего
  отката нет (бэкап — внешний, в туле). Сайт остаётся частично очищенным, тул возвращает ошибку.
- **Рекомендация:** обернуть обработку **каждого файла** в try/catch: при падении — лог + карантин
  файла + продолжить; собрать список упавших файлов в отчёт/`CleanStats`.
- **Уверенность:** подтверждено чтением кода.

### [PIPE-4] 🔴 Bypass · 🟨 Medium · пост-обработка правит даже SKIP_DOM-нутые файлы

- **Сейчас:** блоки удаления ссылок ([:294](src/mastra/cleaners/pipeline.ts:294)+) повторно обходят
  `.html/.htm/.php` **без** проверки `hasServerTags` — то есть редактируют регуляркой и те файлы,
  которым DOM-проходы пропустили.
- **Последствие:** несогласованность: файл с серверными тегами «не чистили», но `<script src>` на
  удалённый metric-файл из него всё же вырезали. Частичная, непредсказуемая правка серверного файла.
- **Рекомендация:** единая политика для серверных файлов (см. PIPE-1); если решено не трогать —
  не трогать и в пост-обработке.
- **Уверенность:** подтверждено чтением кода.

### [PIPE-6] 🟡 Robustness/Perf · 🟨 Medium · сетевые фетчи репина — в горячем цикле, без явного фолбэка

- **Сейчас:** на **каждый** HTML-файл синхронно `await buildCdnReplacements(...)` и
  `buildUnversionedCdnReplacements(...)` ([:193](src/mastra/cleaners/pipeline.ts:193)), а они тянут
  официальный файл по сети ради SRI.
- **Последствие:** офлайн/таймаут/rate-limit CDN → поведение очистки зависит от сети (репин не
  состоится → как тогда классифицируется либа? keep/quarantine?). Плюс последовательные фетчи на
  каждый файл — медленно на многостраничных пачках.
- **Рекомендация:** разобрать в Тир 1 (cdn-detector): явный таймаут/фолбэк, кэш между файлами, и
  детерминированное поведение при недоступности сети (не «тихо оставить»). Здесь — нить на Тир 1.
- **Уверенность:** структура вызовов подтверждена; детали отказа — в cdn-detector.

### [PIPE-5] 🟩 Low · мелочи

- `bytesBefore/After` считают `string.length` (UTF-16 code units), не байты — метрика в отчёте
  неточна для не-ASCII ([:190](src/mastra/cleaners/pipeline.ts:190)).
- Косметический `replace(/\n[ \t]*\n[ \t]*\n+/g, '\n\n')` ([:102](src/mastra/cleaners/pipeline.ts:102))
  может схлопывать значимые пустые строки внутри `<pre>`/`<textarea>`.

## Пробелы в тестах

Нет интеграционных тестов на: SKIP_DOM-ветку (PIPE-1/PIPE-4), промахи regex-удаления ссылок при

## Итог

1. **PIPE-1** — Critical, парно с DOM-1: убрать «тихий SKIP всей очистки».
2. **PIPE-2** — перенести удаление ссылок в DOM, выкинуть 4 копии регекспа.
3. **PIPE-3** — изоляция ошибок по файлу, чтобы один вход не ронял весь прогон.

## ✅ Статус фиксов (C2)

- **PIPE-1 ✅ + DOM-1 ✅ (политика владельца, C2-redo)** — серверный код больше НЕ останавливает
  очистку и НЕ пропускается: `applyHtmlPasses` при `hasServerTags` **вырезает ВСЕ серверные теги**
  (`stripServerTags`) ПЕРВЫМ делом и затем чистит файл полностью (трекеры, CSP, макросы). Владелец
  PHP не использует (своя строка spysecure + форма добавляются на адаптации), поэтому чужой серверный
  код просто удаляется. `stats.serverTagsFilesStripped` + информативная секция в отчёте.
  - **NORM-2 ✅** — следствие закрыто: серверные теги (главный/вторичные, закрытые/незакрытые, ASP)
    вырезаются в pipeline для ВСЕХ файлов, а не только главного в normalize.
  - **2D-4 ✅** — на этих файлах теперь выполняется и `inject-csp` (они больше не пропускаются).
  - Тесты: `html-dom.test.ts` (stripServerTags), `pipeline-quarantine.test.ts` (серверный файл очищен:
    PHP вырезан + трекер удалён), `report.test.ts`.
- **PIPE-2 ✅ + PIPE-4 ✅** — удаление ссылок на удалённые файлы (metric/obfuscated/unversioned/dead)
  переведено с 4 копий regex на ОДИН DOM-проход `stripRefsToDeletedFiles`: резолвит `src`/`href`
  относительно HTML и сверяет с множеством удалённых абсолютных путей → ловит query-string (`x.js?v=3`),
  `../`-пути из подпапок, доп.атрибуты, self-closing (что regex промахивал); чистит и `<link>`/`<img>`,
  не только `<script>`. Серверные файлы (`hasServerTags`) пропускаются → пост-обработка их больше не
  правит (PIPE-4). Тест: `__tests__/strip-refs.test.ts`.
- **PIPE-3 / PIPE-6** — НЕ трогали, остаются 🆕.
