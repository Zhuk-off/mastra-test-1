## verify-runtime.ts (последний рубеж)

### [VR-1] 🔴 Bypass/Workflow · 🟧 High · verify проверяет только ПАССИВНУЮ загрузку → interaction-gated phone-home не ловится

- **Сейчас:** `verifySiteRuntime` делает `goto(networkidle)` + автоскролл + `waitForLoadState`
  ([:92–108](src/mastra/cleaners/verify/verify-runtime.ts:92)). **Кликов нет** (в отличие от
  collect-coverage). `ok = foreignRequests==0 && !pageErrored`.
- **Последствие:** запрос на чужой хост, который срабатывает **по действию** (клик по офферу, шаг
  квиза, сабмит формы, таймер) — во время verify **не триггерится** → `foreignRequests` пуст →
  `ok=true` → «✅ чужих запросов НЕТ» → выгрузка. А кража трафика в арбитраже как раз обычно на
  **клике по офферу**. То есть последний рубеж даёт **ложную уверенность** для самого частого вектора.
- **Рекомендация:** прокликивать CTA/кнопки/шаги (как collect-coverage, но полнее), сабмитить формы
  в preventDefault, ждать таймеры; формулировать вердикт честно: «не звонит **при пассивной загрузке**».
- **Уверенность:** подтверждено чтением кода (нет интеракции).

### [VR-2] 🟡 Robustness · 🟨 Medium · проверяется только одна страница

`pagePath` по умолчанию `index.html` — вторичные `.html` лендинга не проверяются, если тул не позвать
на каждую. Многостраничная воронка: phone-home на шаге 2 не увидим.
- **Рекомендация:** обходить все HTML-страницы сайта (или принимать список).

### [VR-3] 🟢 Security · 🟨 Medium · verify ИСПОЛНЯЕТ чужие запросы (не блокирует)

В отличие от collect-coverage (где `route.abort` для внешних), verify **наблюдает**, но не блокирует —
значит exfil/phone-home реально **уходит** во время проверки (с машины анализа), а редирект-на-загрузке
уведёт `page.goto` на чужую страницу (дальнейшие проверки/скриншот — уже не лендинг). Редирект-навигация
при этом зафиксируется как foreign-request (это плюс). 
- **Рекомендация:** рассмотреть блокировку фактической отправки (record-then-abort) — детект сохранится,
  а данные не утекут; ловить навигацию-редирект отдельно.

### [VR-4] 🟢 Сильная сторона · скриншот fullPage, networkidle-таймаут пойман

`page.screenshot({ fullPage: true })` ([:111](src/mastra/cleaners/verify/verify-runtime.ts:111)) —
в отличие от [VIS-2](3d-coverage-visual.md), здесь полностраничный (правильно). `goto` в try/catch
(таймаут не роняет). Хост-чек по `ALL_TRUSTED_HOSTS` корректен (свой CloudFront/S3 → ок, прочее → аларм).

## tools

### [DL-1] 🟢 Контекст · T-7 ✅ резолвлен + EXT-1 подтверждён

`scripts/download-site.ts` (греп): читает `<base href>` ([:585]) и резолвит URL по нему, затем
**удаляет** `<base>` ([:659]); сторонние ассеты кладёт в `_external/<host>/` ([:92]); переписывает
абсолютные/protocol-relative URL в относительные.
- **T-7:** для сайтов, скачанных нашим тулом, `<base>` уже снят → [2B-1](2b-structural.md) актуален
  в основном для лендингов из **других источников** (ручная копия, wget/httrack-зеркало) — снижает
  частоту, но не отменяет (cleaner всё равно удаляет `<base>` вслепую как fallback).
- **EXT-1:** `_external/<host>/` реально создаётся → [EXT-1](4-css-svg-php-fs.md) релевантна основному пути.

### [DL-2] 🟢 Security · 🟨 Medium · загрузка ИСПОЛНЯЕТ исходный (грязный) лендинг

Скачивание идёт через Playwright-исполнение **оригинальной** (ещё не очищенной) страницы → exfil/редирект
срабатывают во время скачивания, с сетью наружу. Это природа динамического захвата, но стоит зафиксировать
как риск среды анализа (запуск недоверенного кода). Внутренности `download-site.ts` (789 строк) здесь
изучены лишь грепом — полный red-team скрипта вынести отдельно при желании.

### [CST-1] 🟡 Workflow · 🟨 Medium · `clean-site-tool` не отдаёт путь к `clean-report.md`

`execute` возвращает `changelogPath` (и то лишь при js/css>0) и `quarantineDir`, но **не** путь к
`clean-report.md` ([clean-site-tool.ts:91–101](src/mastra/tools/clean-site-tool.ts:91)). А отчёт — это
главный safety-net (см. [REP-1](5a-utils-infra.md)). Агент/человек может не узнать, что его надо
прочитать.
- **Рекомендация:** возвращать `reportPath` всегда; в описании тула — «прочитать clean-report.md».

### [CST-2] 🟩 Low · `noBackup` опасен при destructive-эвристиках

Бэкап по умолчанию включён (хорошо), но `noBackup:true` отключает единственную страховку отката, а
система делает много необратимых удалений (NORM-1 traversal, OBF/MET/COV delete, regex-порча). Стоит
предупреждать о риске `noBackup`.
