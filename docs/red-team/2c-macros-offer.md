# Red Team: Тир 2c — макросы и оффер-ссылки

- [`detect-macros.ts`](src/mastra/cleaners/passes/html/detect-macros.ts) — **твой первый пример**
- [`replace-offer-links.ts`](src/mastra/cleaners/passes/html/replace-offer-links.ts)
- [`offer-detector.ts`](src/mastra/cleaners/utils/offer-detector.ts) (+ `registry/offer-patterns.ts`)

**Контекст твоего примера.** Ты помнил «макросы ищем в HTML и тексте, но не в JS/CSS». По коду +
[docs/macro-and-localization-policy.md](docs/macro-and-localization-policy.md) сейчас сканируются:
атрибуты (`SCAN_ATTRS`+`data-*`), текстовые узлы, **inline `<script>`** (строковые литералы через
AST — без ложных `{a:1}`), **inline `<style>`/`style=""`** (только внутри `url()`). То есть inline
JS/CSS **уже покрыты**. Остаток — ровно по твоей интуиции, но точечно: внешние файлы и синтаксис.

**Сильная сторона (зафиксировать):** AST-скан строковых литералов (а не regex по всему JS) и
url()-only скан CSS — продуманная защита от ложных срабатываний (`{color:red}`, `{a:1}`). Карта
макросов в отчёте решает «удалю и потом не найду». Находки — про границы охвата.

## detect-macros.ts

### [MAC-1] 🟢 Soundness (FN) · 🟨 Medium · внешние `.js`/`.css` не сканируются на макросы (точное место твоего примера)

- **Сейчас:** `detectMacros` — DOM-проход по HTML, видит только **inline** `<script>`/`<style>`/
  атрибуты/текст. Внешние `app.js`, `styles.css` (по `src`/`href`) он не открывает.
- **Последствие:** макрос в внешнем файле — чужой `{clickid}` в `app.js`, наш `{offer}`, которым JS
  строит ссылку, или `url({offerimage})` в внешнем CSS — **не находится, не флагуется, не попадает в
  карту**. Карта макросов в `clean-report.md` неполна → именно «удалю/пропущу и потом не найду, куда
  вставлять» для внешних файлов. Политика-док перечисляет только inline и про внешние молчит — это
  пробел спеки, а не только кода.
- **Рекомендация:** сканировать макросы и во внешних `.js` (AST-строки) и `.css` (`url()`+`content`),
  складывая в ту же карту. Нить **T-8**: подтвердить, что `clean-js`/`clean-css` (Тир 3/4) их сейчас
  НЕ подхватывают (почти наверняка нет — они про трекеры/exfil).
- **Уверенность:** подтверждено чтением кода + политики; финал по clean-js/css — Тир 3/4.

### [MAC-2] 🟢 Soundness (FN) · 🟨 Medium · детектится только `{...}`-синтаксис

- **Сейчас:** `MACRO_RE = /\{[a-zA-Z0-9_:.\-]+\}/g` ([:8](src/mastra/cleaners/passes/html/detect-macros.ts:8)).
- **Последствие:** мимо проходят чужие макросы в иных синтаксисах: `[offer]`, `%offer%`, `{{offer}}`
  (handlebars), `$offer$`, `#macro#`, `{ offer }` (пробелы), `{sub|default}` (пайп). Для **наших**
  макросов это ок (Keitaro — `{}`), но угроза — **чужие** макросы прежнего владельца/сети, которые
  часто используют `[...]`/`%...%`. Они не нормализуются и не флагуются.
- **Рекомендация:** расширить детект на распространённые плейсхолдер-синтаксисы (хотя бы флагом
  «подозрительный плейсхолдер»), либо вынести список синтаксисов в реестр.
- **Уверенность:** подтверждено чтением кода.

### [MAC-3] 🟢 Soundness (FN) · 🟨 Medium · шаблон-скрипты и непарсимый inline-JS пропускаются

- **Сейчас:** для inline `<script>` делается `parseJs`; если не распарсилось → `strings=null` →
  `return` ([:127](src/mastra/cleaners/passes/html/detect-macros.ts:127)).
- **Последствие:** `<script type="text/template">`/`text/x-template` (Handlebars/Underscore — где и
  живут `{{...}}`) парсятся как JS → падают → макросы внутри **не сканируются**. Любой inline-скрипт
  с синтаксисом, который acorn не берёт, — тоже. Комментарий «отработают JS-детекторы» неверен: те
  ищут exfil, а не макросы.
- **Рекомендация:** для `type`-шаблонов (не `application/javascript`) сканировать **сырой текст**
  по MACRO_RE, не через AST. Связано с MAC-2 (там же нужны `{{...}}`).
- **Уверенность:** подтверждено чтением кода.

### [MAC-4] 🟢 Soundness · 🟩 Low · в CSS макрос ищется только в `url()`

- **Сейчас:** `cssUrlMacros` берёт только содержимое `url(...)` ([:14](src/mastra/cleaners/passes/html/detect-macros.ts:14)).
- **Последствие:** макрос в `content:"{offer}"`, в кастомном свойстве `--x:{macro}` или ином CSS-
  контексте не находится. Ограничение url()-only выбрано осознанно (анти-FP на `{color:red}`) —
  компромисс разумный, но `content:` это реальный (редкий) пробел.
- **Рекомендация:** низкий приоритет; при желании добавить скан `content:` со строковым контекстом.
- **Уверенность:** подтверждено чтением кода.

### [MAC-5] 🟢 Soundness · 🟩 Low · авто-`{offer}` только для `<a>`/`<area>`

- **Сейчас:** href, целиком равный макросу, заменяется на `{offer}` только для `a`/`area`
  ([:85](src/mastra/cleaners/passes/html/detect-macros.ts:85)); для `<form action="{macro}">`,
  `<iframe src="{macro}">` → флаг «проверить вручную».
- **Последствие:** редирект-макрос в `form action`/`iframe src` не нормализуется автоматически.
  Приемлемо (реже), но отметить.
- **Уверенность:** подтверждено чтением кода.

## replace-offer-links.ts + offer-detector.ts

### [OFFER-1] 🟡 Robustness/FP · 🟨 Medium · «любая внешняя не-trusted ссылка = оффер»

- **Сейчас:** `looksLikeOfferUrl`: если `host ≠ mainHost` и хост не в `TRUSTED_HOSTS` → `return true`
  (оффер) → `<a href>` заменяется на `{offer}` ([:21–25](src/mastra/cleaners/utils/offer-detector.ts:21)).
- **Последствие:** `TRUSTED_HOSTS` = только lib-CDN + own-assets (соцсети/правовые домены НЕ входят).
  Значит футер-ссылки на facebook/twitter-профиль, на **внешние** terms/privacy, на партнёрский сайт
  → распознаются как оффер → заменяются на `{offer}` → пользователь уходит на оффер вместо политики/
  соцсети. Ложное срабатывание с потерей легитимных ссылок.
- **Рекомендация:** считать оффером только если внешняя ссылка **ещё и** совпала с `OFFER_URL_PATTERNS`
  (или это известный трекер/редирект); вайтлистить типовые соцсети/правовые ссылки.
- **Уверенность:** подтверждено чтением кода.

### [OFFER-2] 🟢 Soundness (FN) · 🟨 Medium · нераспознанный same-host оффер сохраняет ЧУЖОЙ URL

- **Сейчас:** для ссылок на свой домен оффер определяется по `OFFER_URL_PATTERNS`
  ([offer-patterns.ts](src/mastra/cleaners/registry/offer-patterns.ts)); не совпало → не оффер → ссылка
  остаётся как есть.
- **Последствие:** оффер-ссылка с нетиповым параметром → не распознана → в чистом лендинге **остаётся
  URL прежнего владельца/трекера** (утечка провенанса + битый/чужой оффер вместо `{offer}`).
- **Рекомендация:** дополнить паттерны; для подозрительных «click/redirect/go»-подобных путей —
  флагать в отчёт, даже если не заменяем.
- **Уверенность:** подтверждено чтением кода.

### [OFFER-3] 🟩 Low · декодирование и относительные ссылки

- `looksLikeOfferUrl` декодирует только `&amp;` ([:15](src/mastra/cleaners/utils/offer-detector.ts:15))
  — прочие сущности/percent не разворачиваются (паттерн может не сматчиться). Относительные оффер-
  ссылки (`/go/...`) отсекаются ранним `return false` → не заменяются. Низкий приоритет.

## Порядок и пробелы в тестах

- Порядок: `replaceOfferLinks` (#14) → `detectMacros` (#15). Сначала внешние офферы → `{offer}`, затем
  `detectMacros` видит `{offer}` как наш. Плюс `detectMacros` сам делает pure-macro href → `{offer}`.
  Небольшое пересечение, но согласовано.
- Тесты: нет на MAC-1 (внешние файлы), MAC-2 (`[...]`/`%...%`/`{{...}}`), MAC-3 (template-скрипт),
  OFFER-1 (внешняя соцсеть/правовая → ошибочно `{offer}`), OFFER-2 (нетиповой оффер-параметр).

## Итог

1. **MAC-1** — сканировать макросы и во **внешних** `.js`/`.css` (точное место твоего примера); карта
   макросов сейчас неполна. Подтвердить T-8.
2. **MAC-2 + MAC-3** — расширить синтаксис плейсхолдеров и покрыть template-скрипты (там `{{...}}`).
3. **OFFER-1** — не превращать любую внешнюю ссылку в `{offer}` (соцсети/правовые ломаются).
