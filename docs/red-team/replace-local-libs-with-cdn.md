# Red Team: `passes/html/replace-local-libs-with-cdn.ts`

**Роль.** DOM-проход. Берёт карты замен из `ctx` (`cdnReplacements` + `unversionedLibReplacements`,
построены в `pipeline` через `cdn-detector`/`unversioned-cdn-detector`) и подменяет `src`/`href` на
CDN-URL + SRI. Запускается **вторым** проходом — намеренно ДО allowlist, чтобы репинутая либа стала
trusted-CDN и прошла белый список.

---

## Находки

### [RLL-1] 🟡 Robustness · 🟨 Medium · ключ карты — сырой regex-URL, а матч идёт по cheerio-атрибуту

- **Сейчас:** карта замен строится по **точной строке**, выдранной регэкспом из **сырого** HTML
  (`cdn-detector`); здесь же поиск идёт по `$(el).attr('src')`
  ([:20–23](src/mastra/cleaners/passes/html/replace-local-libs-with-cdn.ts:20)) — это значение
  **после** парсинга cheerio (декодированные сущности, нормализация).
- **Последствие:** при расхождении (напр. `src="a.js?x=1&amp;y=2"`: regex дал ключ с `&amp;`,
  cheerio вернёт `&`) → `reps.get(url)` промахивается → репин **тихо не происходит**. Дальше allowlist
  отправит абсолютный фейк-CDN в карантин, а локальную либу оставит — то есть промах **fail-safe**
  (не fail-open), поэтому Medium, а не High. Но репин «молча не сработал» трудно заметить.
- **Рекомендация:** строить ключ карты из того же нормализованного значения (декодировать сущности
  при сборе в `cdn-detector`), либо сверять по нормализованному URL.
- **Уверенность:** подтверждено чтением кода; зависит от наличия сущностей/нормализации в атрибуте.

### [RLL-2] 🟩 Low · ветка «(без SRI: офлайн)» фактически мёртвая

- **Сейчас:** `if (rep.integrity) { … integrity … }`, иначе лог «(без SRI: офлайн)»
  ([:25–28](src/mastra/cleaners/passes/html/replace-local-libs-with-cdn.ts:25)).
- **Последствие:** `cdn-detector` добавляет запись в карту **только при `ok`** (integrity заполнен);
  `unversioned`-путь integrity ставит **всегда**. Значит `rep.integrity` пустым не бывает → ветка
  «без SRI» не исполняется. Комментарий вводит в заблуждение: реальное офлайн-поведение — «репина нет
  вовсе» (запись не создаётся), а не «репин без SRI».
- **Рекомендация:** убрать мёртвую ветку или сделать офлайн-поведение явным (если хотим репинить без
  SRI офлайн — тогда `cdn-detector` должен добавлять запись с пустым integrity).
- **Уверенность:** подтверждено чтением кода (сверено с `cdn-detector` и `unversioned-cdn-detector`).

### [RLL-3] 🟡 Архитектура · 🟨 Medium · полное доверие карте + запуск ДО allowlist

- **Суть:** проход применяет карту замен **с полным доверием** и до белого списка — это и есть
  механизм «репин → trusted → проходит allowlist». Сам проход корректен, но он же **усиливает** любую
  ошибку построения карты: отмывание из [CDN-1](cdn-detector.md) применяется здесь с полным доверием
  и помечается SRI.
- **Рекомендация:** риск закрывается в `cdn-detector` (CDN-1), не здесь. Зафиксировать связь, чтобы
  при фиксе CDN-1 не забыть, что именно этот проход материализует результат.
- **Уверенность:** анализ архитектуры.

---

## Заметки

- `el: any` в `pin` — мелочь типизации, не находка.
- Ставит `crossorigin="anonymous"` вместе с `integrity` — корректно для SRI на кросс-доменном CDN.

## Пробелы в тестах

Нужны: RLL-1 (ключ с `&amp;`/нормализацией → промах репина), RLL-2 (что офлайн-записей в карте не
бывает). Интеграция репина покрыта в `dom-passes`/`cdn-detector` тестах частично.

## Итог

1. **RLL-1** — согласовать ключ карты с cheerio-значением, иначе часть репинов молча не срабатывает.
2. Помнить про **RLL-3 ↔ CDN-1**: этот проход — место, где материализуется отмывание.
