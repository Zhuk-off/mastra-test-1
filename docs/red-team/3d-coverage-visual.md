# Red Team: Тир 3d — coverage и visual

- [`collect-coverage.ts`](src/mastra/cleaners/passes/js-advanced/coverage/collect-coverage.ts) — Playwright, сбор JS-coverage
- [`analyze-coverage.ts`](src/mastra/cleaners/passes/js-advanced/coverage/analyze-coverage.ts) — решение «мёртвый файл → удалить»
- [`verify-visual.ts`](src/mastra/cleaners/passes/js-advanced/verify-visual.ts) — скриншот-диф (pixelmatch)

Оба coverage-файла включаются только при `runCoverage` (по умолчанию **false**), и **удаляют** JS-файлы
с ~0% покрытия. `verify-visual` в `cleanSite` напрямую не вызывается.

**Сильные стороны:** coverage-сервер блокирует внешние запросы (`page.route` → abort non-baseUrl);
автоскролл + клики триггерят часть кода; analyze-coverage **щадит** файлы с event-handlers
(`addEventListener`/`$`/`on*`).

---

## collect-coverage.ts + analyze-coverage.ts

### [COV-1] 🟡 Robustness/FP · 🟧 High · coverage-детекция массово помечает интерактивный код «мёртвым» → удаление

- **Сейчас:** прогон = `goto(networkidle)` + автоскролл + клик по ≤5 `button/[role=button]/.btn/.cta` +
  1с ожидания ([:91–117](src/mastra/cleaners/passes/js-advanced/coverage/collect-coverage.ts:91)).
  Файл с покрытием ≤1% и без event-handlers → `isDead` → `pipeline` его `rm`.
- **Последствие:** лендинг — это **интерактивная воронка**: логика квиза, шагов 2+, попапов, кнопки
  оффера, таймеров выполняется только по конкретному действию. Быстрый авто-прогон их не проходит →
  ~0% покрытия → **удаление нужного кода**. Guard «есть event-handlers» спасает файл, который сам
  вешает обработчик, но **не** утилитный/зависимый файл (`utils.js` с функциями, которые зовёт
  обработчик из `main.js`): у него 0% и нет своих handlers → удаляется → `main.js` ломается
  (висячая ссылка/ReferenceError).
- **Рекомендация:** не **удалять** по coverage, а максимум — флагать кандидатов в отчёт для ручного
  решения (как карантин); учитывать граф зависимостей (импорт/использование), а не только handlers
  в самом файле; гонять полноценный прокликивающий сценарий (как `verify-runtime`), а не 5 кликов.
- **Уверенность:** подтверждено чтением кода; включается опционально, но действие — удаление.

### [ANA-1] 🟡 FP · 🟨 Medium · непарсимый кандидат → считается мёртвым → удаление

- **Сейчас:** `ast = parseJs(...)`; `hasHandlers = ast ? hasEventHandlers(ast) : false`
  ([:58–60](src/mastra/cleaners/passes/js-advanced/coverage/analyze-coverage.ts:58)). Если файл не
  распарсился (PARSE-1, обфускация, хитрый минификат) → `hasHandlers=false` → `isDead=true` → удаление.
- **Последствие:** валидный-но-непарсимый интерактивный файл при 0% покрытия удаляется, хотя мог
  содержать обработчики (их просто не увидели). FP с удалением.
- **Рекомендация:** при `!ast` — не считать мёртвым автоматически (или ставить в карантин с пометкой
  «не проанализирован»).
- **Уверенность:** подтверждено чтением кода.

### [COV-2] 🟡 Robustness · 🟨 Medium · падение coverage роняет весь `cleanSite`

- **Сейчас:** `page.goto(..., { waitUntil: 'networkidle', timeout: 30000 })`. Если страница не достигает
  networkidle (висящее соединение/поллинг) → таймаут → throw; в `pipeline` нет try/catch вокруг
  coverage → `cleanSite` падает целиком ([PIPE-3](pipeline.md)).
- **Последствие:** при `runCoverage:true` патологическая страница прерывает очистку всего сайта на
  этапе coverage.
- **Рекомендация:** обернуть coverage в try/catch (coverage — best-effort, его сбой не должен ронять
  очистку).
- **Уверенность:** подтверждено чтением кода.

### [COV-3] 🟢 Security · 🟩 Low-Med · prefix-обход в статик-сервере

- **Сейчас:** `if (!safePath.startsWith(siteDir)) 403` ([:23](src/mastra/cleaners/passes/js-advanced/coverage/collect-coverage.ts:23)).
- **Последствие:** `startsWith` — префиксная, не граничная проверка: для siteDir `/site` запрос,
  резолвящийся в `/site-evil/x`, проходит (`/site-evil/x`.startsWith(`/site`) === true). Локальный
  эфемерный сервер на 127.0.0.1/random-port, поэтому Low-Med, но это утечка соседней папки.
  Fallback на `index.html` для несуществующих путей тоже выдаёт HTML под видом запрошенного ресурса.
- **Рекомендация:** проверять `path.relative(siteDir, safePath)` без ведущего `..`, или сравнивать с
  `siteDir + path.sep`.
- **Уверенность:** подтверждено чтением кода.

### [COV-4] 🟢 Security · 🟩 Low · исполнение остаточного кода (сеть заблокирована)

Coverage гоняет **остаточный** (не до конца вычищенный) JS лендинга в реальном Chromium. Внешние
запросы режутся `page.route`-abort — это сдерживает exfil/редирект **во время анализа** (хорошая мера).
Остаток: код всё же исполняется (CPU/DOM-side-effects), но сеть наружу закрыта. Отметить как
контролируемый риск.

## verify-visual.ts

### [VIS-1] 🟢 Гигиена · 🟩 Low · не подключён в авто-очистку

`verify-visual` не вызывается из `cleanSite` (pipeline его не импортирует) — это утилита для
verify-тулинга (Тир 5). Находки ниже — про корректность утилиты, не про авто-путь.

### [VIS-2] 🟡 Robustness/FP-negative · 🟨 Medium · сравнивается только вьюпорт (fullPage:false)

`takeScreenshot` снимает фиксированные 1280×800, `fullPage:false`
([:17–19](src/mastra/cleaners/passes/js-advanced/verify-visual.ts:17)). Большая часть лендинга и
**кнопка оффера** обычно ниже первого экрана → их поломка очисткой **не попадёт** в диф. Визуальная
проверка пропускает breakage below-the-fold.
- **Рекомендация:** `fullPage:true` (с равными размерами для pixelmatch) или несколько контрольных
  скриншотов по якорям.

### [VIS-3] 🟡 Robustness · 🟨 Medium · скриншот без блокировки внешних запросов

В отличие от coverage, `takeScreenshot` грузит страницу `goto(networkidle)` **без** `page.route`-блока.
Остаточный код может сделать внешний запрос/редирект во время снимка → «after»-скриншот окажется
страницей-редиректом, а не лендингом → диф-метрика бессмысленна (и потенциальный exfil во время
проверки). Несогласованно с coverage-частью.
- **Рекомендация:** блокировать внешние запросы и при скриншоте; ловить навигацию-редирект.

### [VIS-4] 🟩 Low · pixelmatch требует равных размеров; нет порога решения

`PNG.sync.read` + `pixelmatch` падают/врут при разных размерах baseline/after (сейчас спасает
фиксированный вьюпорт). `verifyVisualDiff` лишь возвращает `diffPercent` — решение «сломалось/нет»
за вызывающим (нормально, но зафиксировать, что порога нет).

## Пробелы в тестах

Нужны: COV-1 (интерактивный/утилитный файл не должен удаляться), ANA-1 (непарсимый → не «мёртвый»),
COV-2 (таймаут coverage не роняет clean), COV-3 (prefix-обход сервера), VIS-2 (below-fold изменение
ловится).

## Итог

1. **COV-1 + ANA-1** — High: не **удалять** по coverage (лендинги интерактивны; утилитные/непарсимые
   файлы ложно «мертвы») — флагать/карантинить, учитывать граф зависимостей.
2. **COV-2** — coverage best-effort: его сбой не должен ронять весь `cleanSite`.
3. **VIS-2 + VIS-3** — визуальная проверка: fullPage + блок внешних запросов, иначе пропускает
   поломку оффера и реагирует на редирект.
